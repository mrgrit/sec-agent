현재(sec-agent v0.1~v0.2 혼합 상태 기준)에서 사용자가 **요구사항을 입력한 이후** 실제로 돌아가는 흐름을 “화면/백엔드/워커” 관점으로 설명.

---

## 1) 요구사항 저장

사용자: 요구사항 텍스트 작성 → **[요구사항 저장]** 클릭
백엔드: `POST /api/projects/{project_id}/requirements`

* DB `requirements`에 한 건 저장

  * `text`: 요구사항 원문
  * `target_id`: 선택된 대상 서버 ID
  * `structured`: (v0.2에선) clarify 답변이 누적될 JSON

결과: “이 프로젝트의 최신 requirement”가 갱신됨.

---

## 2) Clarify(추가 질문 단계)

사용자: **[Clarify]** 클릭
백엔드:

* v0.1: `POST /api/projects/{id}/agent/clarify` (규칙 기반)

  * `target_id` 없으면 “타겟 선택해줘”만 질문
  * 있으면 `done=true`로 끝
* v0.2: `POST /api/projects/{id}/agent/clarify?conn_id=...` (LLM 기반)

  * `target_id` 없으면 동일하게 “타겟 선택”부터 요구
  * 있으면 LLM이 부족한 정보를 질문으로 생성
    예: `mode(IDS/IPS)`, `iface`, `HOME_NET`, `rule_source`, `maintenance_policy` 등

사용자: 질문에 답하고 **[답변 제출]**
백엔드: `POST /api/projects/{id}/agent/clarify/answer`

* `requirements.structured`에 `{field:value}` 형태로 누적 저장

이 단계는 “더 필요한 정보 없음(done=true)”가 될 때까지 반복.

---

## 3) Plan 생성(계획 & TODO 리스트)

사용자: **[Plan 생성]** 클릭
백엔드: `POST /api/projects/{id}/agent/plan`

현재 구현(v0.1 기준)은 대략 이렇게 동작:

1. 최신 requirement를 읽음(원문+target_id)
2. 대상 서버에 SSH 접속해서 **facts 수집**

   * NIC 목록, IP 대역 등
   * “primary iface” 같은 값 추정
3. 그 facts를 바탕으로 **Plan JSON 생성**

   * 가정(assumptions): iface, home_net, mode 등
   * TODO 목록(마일스톤): T1~T6
   * 테스트케이스/롤백 방법 포함(기본)

결과: DB `plans`에 `status=DRAFT`로 저장되고 화면에 Plan이 표시됨.

---

## 4) 사용자 승인(휴먼 검수)

사용자: **[Plan 승인]** 클릭
백엔드: `POST /api/projects/{id}/plan/{plan_id}/approve`

* Plan 상태가 `APPROVED`로 바뀜

이게 중요한 가드레일임: 승인 전에는 “원격 서버 변경 작업”이 실행되지 않도록.

---

## 5) Execute(실제 설치/설정/검증 실행)

사용자: **[실행]** 클릭
백엔드: `POST /api/projects/{id}/agent/execute?plan_id=...`

* DB `tasks`에 작업(Task) 생성: `status=QUEUED`
* Redis RQ 큐에 job enqueue (`run_task(task_id)`)

UI는 동시에:

* `GET /api/tasks/{task_id}/logs/stream` (SSE)로 로그를 실시간 표시

---

## 6) Worker에서 마일스톤 단위로 수행(핵심)

Worker는 `execute_suricata_ids()` 같은 함수로 아래를 순서대로 진행함.

### T1) Precheck & Facts

* OS/NIC/ADDR 정보 수집 명령 실행
* 결과를 “증빙(evidence)” 텍스트로 저장(Artifact)

### T2) 패키지 설치

* `apt update` + `apt install suricata jq`
* stdout/stderr 저장(Artifact)

### T3) 설정 파일 + local.rules 작성

* `/etc/suricata/suricata.yaml` 덮어쓰기
* `/etc/suricata/rules/local.rules` 생성(LOCAL TEST 룰)
* 실행 결과 저장(Artifact)

### T4) 설정 검증

* `suricata -T -c /etc/suricata/suricata.yaml`
* 성공/실패를 exit code로 판정 + 저장

### T5) 서비스 enable/start

* `systemctl enable --now suricata`
* `systemctl is-active suricata` 결과 확인
* 저장

### T6) 스모크 테스트

* `curl http://example.com` 트리거
* `eve.json`에서 “LOCAL TEST” signature 검색
* 찾으면 성공으로 종료
* 못 찾으면 현재 구현에선 보통 `NEEDS_INPUT`로 전환(환경/트래픽/iface 문제 가능)

각 단계에서:

* 로그를 DB `task_logs`로 쌓고(프론트 SSE로 표시)
* 증빙을 파일로 저장하고 DB `artifacts`에 등록함

---

## 7) 완료/실패/추가 입력(상태)

Worker가 끝내면 Task 상태가 다음 중 하나로 종료됨:

* `DONE`: 성공
* `FAILED`: 설치/검증 단계에서 실패
* `NEEDS_INPUT`: 스모크 테스트 미확인 등 “추가 확인 필요”

UI에서 **[상태/아티팩트 갱신]** 누르면:

* `GET /api/tasks/{id}` 로 상태 확인
* 종료 상태면:

  * `GET /api/tasks/{id}/artifacts` 로 증빙 리스트 로드
  * `GET /api/projects/{id}/reports` 로 보고서 로드

---

## 8) 보고서 생성(자동)

성공 시(또는 일부 구현에선 실행 중 생성):

* DB `reports`에 4종 초안 생성

  * 요구사항 분석 / 구현 / 이슈 / 완료 보고서
* UI에서 확인 가능

---

## 한 줄 요약(사용자 관점)

1. 요구사항 저장
2. Clarify로 필요한 정보 채움(필요하면 여러 번)
3. Plan 생성 → 확인 후 승인
4. 실행 → TODO/로그/증빙을 보면서 진행 확인
5. 끝나면 보고서/아티팩트 확인
